Specification and Implementation Plan for a Proof-of-Concept Isolated Web App Browser Utilizing Controlled Frame

1. Introduction and Project Goals

This document outlines the specification and implementation plan for a proof-of-concept (PoC) browser built as an Isolated Web App (IWA) utilizing the <controlledframe> element. The primary objective of this project is to explore the feasibility of reconstructing core browser functionalities—including tabs, an address bar, navigation controls (back, forward, reload), history management, and popup handling—within the IWA security model. This endeavor is not aimed at creating a production-ready browser but rather to serve as an exploratory platform to understand the capabilities and limitations of IWAs and the <controlledframe> API for such a use case.

A key development consideration is the ability to iterate on the user interface (UI) and general application structure efficiently. To this end, the PoC will be designed to operate in two distinct modes:

  1. IWA Mode: Running as a fully packaged and signed Isolated Web App, where the <controlledframe> API is available.
  2. Regular Web App Mode: Running as a standard web application (e.g., from http://localhost or file://), where <controlledframe> is unavailable. In this mode, standard <iframe> elements will be used, and any IWA-specific or <controlledframe> APIs will be mocked to allow for UI development without the overhead of the IWA packaging and deployment cycle.

Extensive debugging capabilities will be integrated to provide clear insights into the application's behavior in both modes.

2. Core Technologies and Concepts

This section details the foundational technologies and APIs that will underpin the PoC browser. A thorough understanding of Isolated Web Apps and the <controlledframe> element is crucial for this project.

2.1. Isolated Web Apps (IWAs)

Isolated Web Apps represent a paradigm for web applications that require a higher level of trust and security than traditional websites.1 They are designed to be secure by design, packaged, isolated, and versioned, enabling access to powerful APIs not typically available to the drive-by web.1

2.1.1. Fundamental Principles

IWAs are built upon the existing web platform but introduce a more stringent security model. This model necessitates a re-evaluation of trust assumptions common in standard web development; for instance, servers and DNS are not explicitly trusted in the same manner.1 All resources required for an IWA to run must be packaged into a Signed Web Bundle (.swbn file).1 This bundle concatenates all assets with an integrity block, allowing the IWA to be securely downloaded, shared, and installed, even offline.1 Consequently, IWAs are well-suited for applications that can operate offline and do not rely on server-rendered pages.1

Navigation within IWAs and access to their resources occur via a new scheme: isolated-app://.1 An IWA's identity is intrinsically linked to the private key used to sign its bundle, meaning each distinct IWA requires a unique signing key.1

2.1.2. Security Model

The security of IWAs is enforced through several mechanisms:

  • Packaging and Signing: All application assets are contained within a Signed Web Bundle, ensuring integrity and authenticity.1 The signing key's public component forms part of the IWA's unique ID.3
  • Isolation: IWAs run in a separate context from regular web browsing and other IWAs. Storage is isolated, preventing data leakage.1
  • Content Security Policy (CSP): IWAs enforce a strict CSP to mitigate risks like cross-site scripting (XSS). Key policies include 1:
  • script-src 'self' 'wasm-unsafe-eval': Allows JavaScript only from the bundle, but Wasm from any source.
  • connect-src 'self' https: wss: blob: data:: Restricts network connections for data fetching.
  • require-trusted-types-for 'script': Mandates Trusted Types for certain DOM manipulations.
  • frame-src 'self' https: blob: data:: Governs embeddable frame sources.
  • object-src 'none' and base-uri 'none': Disallow <object>, <embed>, and <base> elements.
  • default-src 'self': Restricts other resources to the bundle.
  • Permissions Policy: IWAs must explicitly declare required permissions (e.g., geolocation, fullscreen) in their Web App Manifest via the permissions_policy field.1 Permissions are not granted by default.

2.1.3. Manifest and Packaging

An IWA requires a Web App Manifest (manifest.webmanifest), typically located at /.well-known/manifest.webmanifest within the bundle.1 This manifest is similar to that of Progressive Web Apps (PWAs) but includes IWA-specific fields 1:

  • version: A mandatory field specifying the application version (e.g., "1.0.0").1
  • update_manifest_url: An optional but recommended URL pointing to a Web Application Update Manifest, which describes available versions and their bundle locations.1

The packaging process involves gathering all production files and using tools like wbn and wbn-sign (Node.js packages) or Go command-line tools to create the unsigned web bundle and then sign it with an Ed25519 or ECDSA P-256 key pair.1

2.1.4. Development and Debugging Workflow

Developing and testing IWAs involves specific steps 1:

  • Enable IWA developer mode in Chrome via the chrome://flags/#enable-isolated-web-app-dev-mode flag.
  • Install and manage IWAs through chrome://web-app-internals. This interface allows installation from a development proxy (for unbundled development) or directly from a signed .swbn file.
  • Chrome DevTools are available for debugging the IWA's JavaScript, DOM, and network activity.

The necessity of bundling and signing for each iteration in IWA mode can impede rapid development cycles, particularly for UI adjustments and general logic changes. The proposed dual-mode architecture, with a "Regular Web App Mode" employing <iframe> and mocked APIs, directly addresses this potential bottleneck. By allowing developers to work on significant portions of the application without the IWA-specific overhead, this approach aims to enhance developer experience and iteration speed.

2.2. The <controlledframe> Element

The <controlledframe> HTML element is a cornerstone of this PoC, providing the mechanism for embedding and controlling web content within the IWA shell.8

2.2.1. Purpose and Core Capabilities

The <controlledframe> element is designed exclusively for use within Isolated Web Applications.5 Its primary purpose is to allow an IWA to embed arbitrary web content with a greater degree of control than is possible with a standard <iframe>.

Key capabilities include:

  • Embedding Restricted Content: It can embed content even if that content uses X-Frame-Options or Content Security Policy directives to prevent embedding in an <iframe>.5 The specification notes it "should allow embedding all content without express permission from the embedded site".8
  • Top-Level Browsing Context Illusion: Content loaded within a <controlledframe> operates in what appears to it as a new top-level browsing context.5 This means the embedded content cannot typically detect that it is framed using common JavaScript checks like window.top!== window.self, as "the embedded navigable appears as a top-level traversable with a null parent" 8 and "window should match window.parent and window.top".5
  • Enhanced Embedder Control: The IWA shell (the embedder) gains significantly more control over the embedded content's lifecycle, navigation, and permissions compared to an <iframe>.5

2.2.2. Key API Features for Browser Emulation

The <controlledframe> element exposes a rich API through its attributes, methods, and events, many of which are directly applicable to emulating browser features:

  • Element Attributes:
  • src: A string specifying the URL of the content to be loaded into the frame.5 This URL must use http:, https:, or data: schemes. Notably, it cannot be an isolated-app: URL, preventing an IWA from embedding another IWA or itself via <controlledframe>.5
  • partition: A string attribute that defines a separate storage partition for the content within the <controlledframe>.5 This is critical for isolating cookies, localStorage, IndexedDB, and other storage mechanisms on a per-frame basis. Partitions can be in-memory (cleared when the element is destroyed or the IWA closes) or persistent if prefixed with persist: (e.g., partition="persist:myframepartition").5 The partition attribute is fundamental for implementing tabbed browsing. Each tab, represented by a <controlledframe>, can be assigned a unique persistent partition. This effectively creates isolated storage environments for each "tab," mimicking how real browsers handle sessions and data for different tabs, even those open to the same website. For example, a user could be logged into different accounts on the same service in two separate tabs.
  • Navigation Methods: These methods allow the IWA shell to programmatically control the browsing history of the embedded content 8:
  • back(): Navigates to the previous entry in the session history. Returns a Promise resolving to a boolean indicating success.
  • forward(): Navigates to the next entry in the session history. Returns a Promise resolving to a boolean.
  • go(relativeIndex): Navigates to a specific entry in session history relative to the current one (e.g., go(-1) is like back(), go(0) reloads). Returns a Promise.
  • reload(): Reloads the current content.
  • stop(): Stops the current page load.
  • canGoBack(): Returns a boolean indicating if there's a previous history entry.
  • canGoForward(): Returns a boolean indicating if there's a next history entry. These methods and properties are direct analogues for browser navigation buttons and their state management.
  • Content Interaction Methods:
  • executeScript(details): Allows the IWA shell to inject and execute JavaScript code within the context of the <controlledframe>.5 The details object can specify either a code string or a list of files to inject. This method returns a Promise that resolves with the result of the script execution.
  • insertCSS(details): Injects CSS styles into the embedded content.8
  • addContentScripts(contentScriptList): Adds content scripts that persist across navigations within that frame, similar to browser extension content scripts.8
  • removeContentScripts(scriptNameList): Removes previously added content scripts.8
  • Event Handling: The <controlledframe> element emits various events that the IWA shell can listen to for monitoring and reacting to the state of embedded content 8:
  • Navigation Lifecycle Events: onloadstart, onloadcommit, oncontentload, onloadstop, onloadabort, onloadredirect. These allow tracking the full lifecycle of a navigation within the frame. onloadcommit, for example, is useful for knowing when a navigation (including redirects) has finalized and the URL is stable.
  • onconsolemessage: Fired when JavaScript within the <controlledframe> calls console logging methods (e.g., console.log).8 This allows the IWA shell to capture and display these messages, crucial for debugging.
  • onpermissionrequest: Fired when the embedded content requests a permission, such as geolocation, camera, or microphone access.5 The IWA shell receives this event and can decide to allow() or deny() the request. The IWA can only grant permissions it itself possesses.
  • onnewwindow: This event is critical for handling popups. It fires when the embedded content attempts to open a new window, typically via window.open().5 The event details usually include the target URL and requested window features. The IWA shell can intercept this, prevent the default browser behavior, and instead create a new <controlledframe> (perhaps in a new tab or modal) to host the popup content.
  • ondialog: Fired when the embedded content tries to display standard dialogs like alert(), confirm(), or prompt().8 The IWA shell can intercept these and provide its own UI.
  • State/Configuration Methods:
  • getAudioState(): Returns a Promise resolving to a boolean indicating if audio is playing in the frame.
  • isAudioMuted(): Returns a Promise resolving to a boolean indicating if audio is muted.
  • setAudioMuted(mute): Mutes or unmutes audio in the frame.
  • getZoom(): Returns a Promise resolving to the current zoom factor.
  • setZoom(zoomFactor): Sets the zoom factor.
  • clearData(options, types): Clears browsing data (cache, cookies, etc.) for the frame's partition.8
  • Capture Method:
  • captureVisibleRegion(options): Captures a screenshot of the visible area of the <controlledframe> content, returning a data URL.5

2.2.3. Security and Isolation Properties

The <controlledframe> element is designed with strong security and isolation guarantees:

  • Process Isolation: Content within a <controlledframe> renders in a separate operating system process from the IWA shell's process.5 This is a significant feature mirroring modern browser architecture.10 It enhances stability, as a crash or hang in the content of one "tab" (one <controlledframe>) will not bring down the entire IWA browser shell. It also provides a security boundary.
  • Storage Partitioning: As discussed with the partition attribute, each <controlledframe> can have its storage (cookies, localStorage, IndexedDB, etc.) isolated.5 The specification states it will "store Controlled Frame site data in a 'storage user agent' apart from the default and IWA storage user agents".5 This prevents data leakage between different frames and between a frame and the IWA shell itself.
  • No Direct DOM Access: For security reasons, the IWA shell cannot directly access or manipulate the DOM of the content loaded within a <controlledframe>, and vice-versa.5 Communication and interaction are mediated through the executeScript method, addContentScripts, and events. While direct DOM access is blocked, these mechanisms act as a controlled bridge. The IWA shell can use executeScript to run JavaScript within the frame's context, and that script can then interact with the frame's DOM. This is analogous to how browser extensions use content scripts to interact with web pages.

2.2.4. Comparison with <iframe>

The <controlledframe> element offers distinct advantages over the standard <iframe> for the purpose of building a browser-like application within an IWA:

  • Embedding Capabilities: <controlledframe> can embed sites that use X-Frame-Options or CSP to block <iframe> embedding.5
  • Top-Level Context: Content within <controlledframe> generally believes it is in a top-level window, unlike <iframe> content which can easily detect its embedded status.5
  • Isolation: <controlledframe> provides superior process isolation and more granular storage partitioning.5
  • Control APIs: <controlledframe> offers a richer set of APIs for navigation control, script injection, and event handling relevant to browser functions.5
  • Availability: <iframe> is a standard HTML element available in all web contexts. <controlledframe> is an IWA-specific feature and is undefined outside of an IWA environment.5

3. Architectural Design and Implementation Strategy

The architecture of the PoC browser will be designed to support both IWA and regular web app modes, focusing on modularity and clear separation of concerns.

3.1. Overall Application Architecture

The browser shell—the UI comprising the address bar, tabs, navigation buttons, etc.—will be implemented as a Single Page Application (SPA). This approach allows for dynamic UI updates as the user interacts with the browser, without requiring full reloads of the IWA itself. Standard DOM manipulation techniques will be employed, potentially augmented by a lightweight UI library if deemed necessary, though vanilla JavaScript is preferred for a PoC. The Navigation API 11 could be considered for managing complex states within the shell itself, should the shell's internal navigation (e.g., between a main browsing view and a history or settings view) become intricate.

A fundamental architectural principle is the clear separation between the Shell Logic and Content Rendering:

  • Shell Logic: This encompasses all the JavaScript code bundled within the IWA that manages the browser's UI, tab states, history persistence, user settings, and the orchestration of content view instances (which will be <controlledframe> or <iframe> elements).
  • Content Rendering: This is delegated to instances of <controlledframe> (in IWA mode) or <iframe> (in regular web app mode). These elements are responsible for loading, displaying, and interacting with external web content.

Communication between the shell and the content frames will be primarily event-driven. For instance, navigation requests from the address bar will trigger actions on the active content frame, while events like onnewwindow from a <controlledframe> will notify the shell to handle popup creation.

3.2. Dual-Mode Implementation Strategy (IWA vs. Regular Web App)

The dual-mode strategy is a practical concession to the development lifecycle of IWAs. The goal is to enable rapid iteration on the UI and general application logic without the repetitive overhead of bundling, signing, and installing the IWA for every minor change [User Query].

  • IWA Mode:
  • Operates when the application is loaded via the isolated-app:// scheme.
  • Utilizes the <controlledframe> element for rendering all web content.
  • Has full access to IWA-specific APIs and adheres to the IWA security model.
  • Requires enabling IWA developer mode in the browser and installing the packaged application.1
  • Regular Web App Mode:
  • Operates when the application is loaded via http://localhost (during development) or file:// schemes.
  • Uses standard <iframe> elements to embed web content.
  • Crucially relies on a Mocking Layer. This layer will consist of JavaScript modules that provide mock implementations of the <controlledframe> element's API and any IWA-environment-specific properties or behaviors that the shell logic depends on. For example, a MockControlledFrame class would attempt to mimic methods like src (by setting iframe.src), back() (by calling iframe.contentWindow.history.back()), executeScript() (perhaps using postMessage for limited cases or attempting direct injection which would often fail for cross-origin content), and emit simulated events like onnewwindow (by attempting to override iframe.contentWindow.open).
  • It's important to acknowledge that <iframe> elements have inherent limitations compared to <controlledframe>: they cannot bypass X-Frame-Options or certain CSP directives, and the embedded content can easily detect it is framed. The mock layer cannot perfectly replicate all <controlledframe> behaviors but should aim for functional equivalence sufficient for the shell's logic to operate.

At startup, the application will detect its execution environment. This detection could be based on checking the window.location.protocol for isolated-app: or by attempting to create a <controlledframe> element and checking for its specific API ('src' in document.createElement('controlledframe')). Based on this detection, the application will dynamically choose to use either the real <controlledframe> elements or the <iframe>-based mock wrappers.

The UI components of the browser shell (tabs, address bar, buttons) will be implemented using standard web technologies (HTML, CSS, JavaScript) and designed to be largely independent of the underlying content rendering mechanism. This necessitates an abstraction layer. A well-defined interface, perhaps an IContentView class or prototype, should be established for the "content view" component. The rest of the shell logic (tab management, navigation control) will interact with this abstracted interface, which internally delegates calls to either a real <controlledframe> wrapper or the <iframe>-based mock wrapper. This design simplifies the shell's logic, makes the mode-switching cleaner, and promotes code reusability.

The table in section 2.2.4 ("<controlledframe> vs. <iframe> for PoC Browser Features") outlines the practical differences developers will face and highlights what aspects of <controlledframe> are challenging or impossible to perfectly mock with <iframe>. This table is crucial for guiding the implementation of the mock layer and setting realistic expectations for the regular web app mode.

3.3. Component Breakdown (Conceptual Modules)

The application can be conceptually broken down into the following modules:

  • BrowserShellUI: Manages the overall visual structure of the browser, including the address bar, navigation buttons (back, forward, reload, stop), and the tab bar. It captures user input from these elements and delegates actions to other relevant modules.
  • TabManager: Responsible for the lifecycle of tabs. This includes creating new tabs, destroying closed tabs, activating selected tabs, and tracking the state of each tab. Each logical tab in the UI will correspond to an instance of a content view (either <controlledframe> or <iframe>). It will maintain a list or collection of all open tabs.
  • NavigationController: Handles navigation requests originating from the address bar or UI buttons (e.g., a "home" button). It instructs the active tab's content view to navigate to a new URL or perform history actions (back, forward, reload). It also listens to navigation lifecycle events from the content view to update the UI (e.g., loading indicators, displayed URL in the address bar, tab titles).
  • HistoryService: Captures navigation events (specifically, committed navigations with their URL and title) from the NavigationController or TabManager. It persists these history entries using a suitable storage mechanism, likely IndexedDB. It will provide methods to query this history, which could be used for a dedicated history page or to inform back/forward list data (though the latter is often directly managed by the frame's canGoBack/canGoForward states).
  • PopupService: Listens for onnewwindow events (or their mocked equivalents) emitted by content views. This service will manage the creation and display of popups, likely by instructing the TabManager or BrowserShellUI to create a new, possibly simplified, tab or a modal-like window structure that hosts its own new <controlledframe> (or <iframe>) instance for the popup's content.
  • ContentViewFactory: A factory module responsible for instantiating content view wrappers. Based on the detected operating mode (IWA or regular web app), it will create and return either a wrapper around a real <controlledframe> element or a wrapper around an <iframe> element that includes the mocking logic. Both types of wrappers should adhere to a common IContentView interface to ensure consistent interaction from other modules like TabManager and NavigationController.
  • StorageService: Provides an abstraction layer for interacting with client-side storage. This module will encapsulate the logic for using IndexedDB (for structured data like history and bookmarks) and localStorage (for simpler settings like a homepage URL).
  • DebuggingService: A centralized utility for logging and diagnostics. It will provide standardized methods for other modules to log messages. A key function will be to listen to onconsolemessage events from <controlledframe> instances and forward these messages to the main IWA's DevTools console, aiding in debugging the content of frames.

4. Feature Specifications and Implementation Plan

This section details the specific browser features to be implemented in the PoC and outlines their implementation approach.

4.1. Browser Shell UI

  • Address Bar:
  • An HTML <input type="text"> element will serve as the URL entry field.
  • It will dynamically display the URL of the currently active tab. This update should occur when a tab is switched or when a navigation within the active tab commits to a new URL.
  • A visual indicator (e.g., a spinner icon or subtle animation) should be displayed within or near the address bar when the active tab is in a loading state.
  • Pressing the 'Enter' key while the address bar has focus will trigger a navigation attempt in the active tab to the URL entered. Input validation (e.g., ensuring a scheme is present or prepending http://) might be considered.
  • Navigation Controls:
  • Standard browser navigation buttons (Back, Forward, Reload, Stop) will be implemented as HTML <button> elements. An optional "Home" button can be included, navigating to a configurable homepage URL.
  • The enabled/disabled state of the Back and Forward buttons must dynamically reflect the navigation state of the active content view. This will use activeControlledFrame.canGoBack() and activeControlledFrame.canGoForward() in IWA mode, or the mocked equivalents based on iframe.contentWindow.history in regular mode.
  • The 'Stop' button, when clicked, should invoke the stop() method on the active <controlledframe> or window.stop() on the active <iframe>'s content window to halt the current page load.
  • Tab Bar:
  • The tab bar will display a list of all open tabs. Each tab representation in the UI should ideally show the page title and potentially its favicon (fetching favicons is an optional enhancement).
  • A dedicated button (e.g., a '+' icon) will allow users to create a new tab.
  • Each tab representation in the UI will have a close button (e.g., an 'x' icon) to close that specific tab.
  • The currently active tab must be visually distinct from inactive tabs (e.g., different background color, border).

4.2. Tab Management

  • Creating New Tabs:
  • When the 'New Tab' button is clicked, the TabManager will instantiate a new content view using the ContentViewFactory. This will be a <controlledframe> in IWA mode or an <iframe> in regular mode.
  • Each new tab must be assigned a unique identifier. If persistent partitions are used for <controlledframe> instances (e.g., persist:tab-<uuid>), a unique partition string must also be generated and assigned. This ensures that each tab has its own isolated storage for cookies, localStorage, etc., when running in IWA mode.
  • New tabs can be configured to load a default page, such as "about:blank", a simple bundled HTML page (e.g., a speed dial or new tab page), or a user-configurable homepage.
  • The TabManager will add the new tab object (containing its ID, content view instance, and state) to its internal list of tabs, and the BrowserShellUI will update the visual tab bar. The new tab typically becomes the active tab.
  • Switching Tabs:
  • Clicking on a tab in the tab bar will designate it as the active tab.
  • The TabManager will instruct the BrowserShellUI to visually update the active tab. This involves showing the <controlledframe> or <iframe> element corresponding to the newly activated tab and hiding the elements of all other tabs.
  • The address bar URL, navigation button states (Back/Forward enabled/disabled), and potentially the window title should be updated to reflect the state of the newly active tab.
  • Care should be taken to manage focus, ensuring that keyboard focus is appropriately transferred to the new active content frame or the address bar, depending on user expectations.
  • Closing Tabs:
  • When a tab's 'Close' button is clicked, the corresponding <controlledframe> or <iframe> element must be removed from the DOM.
  • Any necessary cleanup or disposal methods available on the content view wrapper or the underlying frame element should be invoked. For <controlledframe>, there isn't an explicit dispose() method in the current API documentation 8, so removal from DOM and releasing references might be sufficient.
  • The TabManager will remove the tab from its internal tracking, and the BrowserShellUI will update the visual tab bar.
  • If the last tab is closed, the browser might be configured to open a new default tab or exhibit other defined behavior.
  • Background Tab State (Optional for PoC):
  • For a more advanced PoC, visual indicators could be added to background tabs, such as an icon indicating audio is playing (using controlledFrame.getAudioState() 8) or if there are unread notifications (if a notification system were implemented).

Efficient management of <controlledframe> instances is an important consideration. Since each <controlledframe> is designed to run in a separate process 5, creating a large number of them could lead to significant resource consumption, mirroring challenges in real-world browser development.10 While simply showing/hiding DOM elements might be adequate for a PoC with a few tabs, strategies like "sleeping" background tabs (e.g., calling stop() on inactive frames, though the API doesn't offer more sophisticated power-saving modes) or more aggressive lifecycle management might be needed for applications intending to handle many tabs. This PoC should monitor resource usage as tab count increases.

4.3. Navigation

  • Loading URLs:
  • When a URL is entered into the address bar and submitted, the NavigationController will instruct the active tab's content view to load that URL. This is typically done by setting the src attribute of the <controlledframe> or <iframe>.
  • Link clicks within the content of a <controlledframe> or <iframe> will be handled by the frame itself by default, triggering standard navigation within that frame.
  • Back/Forward Navigation:
  • The 'Back' button will invoke activeControlledFrame.back() or activeControlledFrame.go(-1).8 In regular mode, this maps to iframe.contentWindow.history.back().
  • The 'Forward' button will invoke activeControlledFrame.forward() or activeControlledFrame.go(1).8 In regular mode, this maps to iframe.contentWindow.history.forward().
  • Reload:
  • The 'Reload' button will invoke activeControlledFrame.reload().8 In regular mode, this maps to iframe.contentWindow.location.reload().
  • Page Load Lifecycle Events:
  • In IWA mode, the shell will listen to <controlledframe> navigation events such as onloadstart, onloadcommit, oncontentload, and onloadabort.8
  • These events will be used to update the UI:
  • onloadstart: Show loading indicator.
  • onloadcommit: Update the address bar with the final URL (which might differ from the initial src due to redirects; the event details should provide this URL). Update the tab title (which might require executeScript if the title isn't directly available from event details). Record the navigation for history.
  • oncontentload: Could be used to signify the main content has loaded.
  • onloadabort: Clear loading indicator, potentially show an error page.
  • In regular mode, the <iframe>'s load event is the primary indicator of a completed load. Simulating the finer-grained events of <controlledframe> would be complex and likely not necessary for the PoC's core goals beyond updating the URL and loading state.
  • Navigation API (window.navigation):
  • The window.navigation API 11 is primarily designed for managing navigation within SPAs. For this PoC, its main utility would be for managing the shell's own state if the shell UI becomes complex (e.g., navigating between a "browsing view," a "history page view," and a "settings view" all within the single IWA page).
  • Navigation within the content displayed in a <controlledframe> is managed by the <controlledframe>'s own methods and its internal browsing context. The Navigation API, if used by the embedded content itself, will function as it normally would on the web, independently of the shell's use of the API.
  • This distinction is important: there's content navigation (handled by <controlledframe>) and potentially shell navigation. The PoC primarily focuses on the former. If the shell's UI evolves to include distinct internal "pages," the Navigation API would be the appropriate tool for managing those transitions, as described for SPAs.13

4.4. History Management

  • Capturing History Entries:
  • Upon successful navigation completion in a tab (e.g., on the onloadcommit event from a <controlledframe>, or the load event from an <iframe>), the NavigationController or TabManager will capture the necessary information.
  • This information should include the final URL of the page and its title. The title might need to be retrieved using executeScript if it's not directly available as part of the navigation event details (e.g., cf.executeScript({ code: 'document.title' })).
  • A history entry object will be created, e.g., { url: string, title: string, timestamp: Date, tabId: string (optional, for context) }.
  • Storage:
  • History entries will be persisted using IndexedDB, managed via the StorageService module. Standard IndexedDB APIs will be used.14
  • An object store specifically for history entries should be created within an IndexedDB database.
  • This object store should be indexed by timestamp (for chronological sorting) and potentially by url (for quick lookups or de-duplication if desired, though browser history typically allows multiple entries for the same URL).
  • Displaying History (Optional UI):
  • While a full-featured history page is optional for the PoC, a simple page or view could be implemented within the IWA shell. This view would query IndexedDB via the HistoryService and list the history entries.
  • Clicking on a history entry in this view should load the corresponding URL in a new or active tab.
  • More critically for the PoC's core functionality, the persisted history data underpins the back/forward navigation capabilities, although the actual traversal is handled by the frame's internal history stack and canGoBack/canGoForward methods.

4.5. Popup Handling (onnewwindow)

  • Intercepting Popups:
  • In IWA mode, the shell will attach an event listener for the onnewwindow event to each <controlledframe> instance.5 The event details are expected to provide the target URL for the new window and potentially other parameters like window name or features..23
  • In regular web app mode (using <iframe>), intercepting window.open() calls from within an <iframe> is more complex. If the <iframe> is sandboxed with allow-popups but not allow-popups-to-escape-sandbox, popups might be constrained. A common approach to intercept is to attempt to override iframe.contentWindow.open. However, this is subject to browser security restrictions and may not always be reliable, especially for cross-origin <iframe>s. For the PoC's mock, a simpler strategy might be to log the attempt, or, for development purposes, allow the real browser to open a new tab/window, acknowledging this differs from the IWA mode's controlled behavior.
  • Creating Popup "Windows":
  • When the onnewwindow event fires in IWA mode, the PopupService should:
  1. Prevent the default action of the event if the API allows (e.g., via event.preventDefault() or a method on an event-provided request object). This stops the browser from trying to open a new OS-level window.
  2. Create a new UI element within the IWA shell to represent the popup. This could be a new, simplified "tab" managed by the TabManager, or a modal-like overlay component.
  3. This new UI element will host a new <controlledframe> instance, created via the ContentViewFactory.
  4. The src attribute of this new <controlledframe> will be set to the target URL obtained from the onnewwindow event details.
  5. The PoC may optionally attempt to respect requested window features (like size) if they are provided by the event and if the UI design for popups can accommodate them.

The onnewwindow event mechanism 5 is the designated pathway for an IWA to manage popup requests originating from its embedded content. Instead of allowing a true new OS window (which would break the isolation model and reduce control), the IWA creates another managed <controlledframe> within its own UI. This maintains the "browser-within-an-app" paradigm and ensures all browsing content remains under the IWA's control and subject to its policies. This is a key differentiator in how an IWA-based browser would handle popups compared to a native browser or standard web page. While general popup handling examples exist 17, the <controlledframe> approach is specific to this IWA context.

4.6. (Optional PoC Feature) Basic Bookmarking

  • UI: A simple button, perhaps in the address bar or alongside navigation controls, to "bookmark" the current page of the active tab. A separate, simple page or view within the IWA shell could list saved bookmarks.
  • Logic: When the bookmark button is clicked, the shell would retrieve the URL and title of the active tab's current page. A bookmark entry object, e.g., { url: string, title: string, timestamp: Date, id?: string }, would be created.
  • Storage: Bookmarks would be persisted using either IndexedDB (for more structured storage or larger numbers of bookmarks) or localStorage (for a very simple implementation with a small number of bookmarks) 14, managed via the StorageService. Clicking a bookmark in the manager view would load the URL in a new or active tab.

5. API Usage and Integration Details

This section specifies how the various APIs will be used to implement the PoC's features, differentiating between IWA mode and regular web app mode.

5.1. IWA-Specific API: <controlledframe>

  • Instantiation: A <controlledframe> is created programmatically using document.createElement('controlledframe'). Its src and partition attributes are then set, and the element is appended to the DOM, typically within a container representing a tab.
    JavaScript
    // Example: Creating a new controlledframe for a tab
    const frame = document.createElement('controlledframe');
    frame.id =     `cf-${tabId}`;
    frame.src = initialUrl;     // e.g., 'about:blank' or a new tab page
    frame.partition =     `persist:tab-${tabId}`; // Unique persistent partition
    tabContentContainer.appendChild(frame);
  • Navigation:
  • Loading a new URL: controlledFrameElement.src = newUrl;
  • Reloading: controlledFrameElement.reload();
  • Going back: controlledFrameElement.back(); or controlledFrameElement.go(-1);
  • Going forward: controlledFrameElement.forward(); or controlledFrameElement.go(1);
  • Stopping load: controlledFrameElement.stop();
  • State Checking for UI Updates:
  • canGoBack(): const canNavBack = await controlledFrameElement.canGoBack(); backButton.disabled =!canNavBack; (Note: spec shows canGoBack as synchronous boolean 8, but usage with await implies it might be or become async, or the example was illustrative. Verify against final spec).
  • canGoForward(): Similar to canGoBack().
  • Scripting:
  • controlledFrameElement.executeScript({ code: 'alert(document.title);' }).then(results => /*... */);
  • controlledFrameElement.executeScript({ files: ['/scripts/my-injection-script.js'] }).then(results => /*... */); 5
  • Event Handling (illustrative, actual event object properties need to be used from the specification 8):
  • controlledFrameElement.addEventListener('loadcommit', (event) => { console.log('Navigation committed:', event.url); /* Update address bar, history */ });
  • controlledFrameElement.addEventListener('onnewwindow', (event) => { console.log('New window request:', event.targetUrl, event.windowName); popupService.handleNewWindow(event); });
  • controlledFrameElement.addEventListener('onpermissionrequest', (event) => { console.log('Permission requested:', event.permission); if (iwaHasPermission(event.permission)) { event.allow(); } else { event.deny(); } }); 5
  • controlledFrameElement.addEventListener('consolemessage', (event) => { console.log(:, event.message); }); 8

5.2. Standard Web APIs for Shell

The IWA shell itself, being a web application, will leverage standard web APIs for its functionality:

  • DOM APIs: Standard methods like document.getElementById(), document.createElement(), element.appendChild(), element.addEventListener(), etc., will be used for constructing and manipulating the browser shell's UI.
  • Navigation API (window.navigation) 11:
  • If the shell UI evolves to have its own internal "pages" or distinct views (e.g., a main browsing interface, a history viewer page, a settings page), the Navigation API is the modern, recommended way to manage these SPA-like transitions.
  • Example: navigation.navigate('/shell/settings'); could change the shell's UI to display a settings panel.
  • An event listener navigation.addEventListener('navigate', event => { /*... */ }); would handle these internal shell navigations, potentially calling event.intercept() to update the shell's DOM.
  • navigation.currentEntry.getState() and navigation.updateCurrentEntry({ state: newState }) can be used to manage state associated with the shell's current view (e.g., scroll position on the history page).
  • History API (window.history) 11: As a simpler alternative for managing shell view states in the PoC (if the Navigation API is deemed overly complex for initial shell states), history.pushState(), history.replaceState(), and the popstate event could be used. However, the Navigation API is generally preferred for new SPA development due to its more robust features.13
  • Storage APIs:
  • IndexedDB: This will be the primary choice for storing structured data like browsing history and bookmarks due to its capacity for larger datasets and querying capabilities.14 It will be accessed via the StorageService abstraction layer.
  • Limitations: IndexedDB operations are asynchronous and more complex to use than localStorage. While quotas are generally large (potentially gigabytes, depending on available disk space and browser 14), they are not unlimited, and QuotaExceededError should be handled.
  • localStorage: Suitable for storing small, simple key-value pairs, such as user preferences (e.g., homepage URL, theme settings).14
  • Limitations: localStorage is synchronous, which can block the main thread if used excessively or with large data. Its storage quota is significantly smaller than IndexedDB (around 5-10MB per origin).14
  • The storage used by the IWA shell (its own localStorage and IndexedDB) is distinct and isolated from the storage used by content loaded within <controlledframe> elements (which is further isolated by the partition attribute) and also from the user's regular browser storage for the same origins when visited outside the IWA.1 This multi-layered storage isolation is a core security and privacy benefit of the IWA model. The IWA shell has its own dedicated storage for its operational data, while each embedded piece of content has its own, potentially partitioned, storage.
  • Fetch API: If the IWA shell needs to make its own network requests (e.g., to fetch favicons for tabs, or to check an update manifest URL, assuming the IWA's CSP allows such connections via connect-src), the Fetch API will be used.

5.4. chrome.* APIs (e.g., chrome.storage, chrome.runtime)

The chrome.storage and chrome.runtime APIs are primarily designed for Chrome Extensions and Chrome Apps.19 While an IWA can be configured to communicate with an owned companion Chrome Extension (by declaring the IWA's isolated-app:// origin in the extension's externally_connectable manifest key 1), the IWA shell itself does not directly gain access to these chrome.* APIs for its own internal use in the same way an extension's background script or content script would.

For this PoC browser, the IWA shell will not rely on chrome.storage or chrome.runtime for its own persistence or runtime information needs. Instead, it will use standard Web Storage APIs (IndexedDB and localStorage) as described in section 5.2. Communication with companion extensions is considered out of scope for this core PoC, which focuses on the browser functionalities achievable with IWA and <controlledframe> primitives. This approach of using standard web APIs for the shell's internal data management promotes portability and aligns with the "web app" nature of IWAs, even with their enhanced privileges and isolation.

6. Debugging and Diagnostics Plan

A robust debugging strategy is essential for understanding the behavior of the PoC, especially given the interplay between the IWA shell and the <controlledframe> content, and the dual-mode operation.

6.1. Comprehensive Logging

A Logger module will be implemented within the IWA shell. This module will provide a centralized way to log messages with varying severity levels (e.g., INFO, WARN, ERROR).

Key events and state changes to log include:

  • Application startup, including detected mode (IWA/Regular).
  • Tab lifecycle: creation, activation, closing, associated <controlledframe> partition ID.
  • Navigation attempts: URL, source (address bar, link click, popup).
  • Navigation outcomes: success, failure, errors.
  • <controlledframe> lifecycle events (onloadstart, onloadcommit, etc.).
  • Errors caught in try...catch blocks. Logs should be prefixed for clarity (e.g., ``, [NavigationCtrl], [CF-View:<tabId>]). Standard console.log(), console.warn(), and console.error() methods will be used.

6.2. Leveraging <controlledframe> Debugging Features

A significant aid to debugging content within <controlledframe> elements is the onconsolemessage event.8 The IWA shell will listen for this event on each <controlledframe> instance. When a message is received, the DebuggingService will forward it to the main IWA's DevTools console. The output will be prefixed to indicate which tab (and thus, which <controlledframe>) originated the message. For example: : Hello from embedded page!. This allows developers to see console output from the sandboxed content directly within the shell's debugging environment. The specification notes: "Console messages within a nested browsing context fire an event that the embedder can choose to display".9

6.3. Chrome DevTools for IWAs

When an IWA is installed in developer mode (e.g., via chrome://web-app-internals using the dev proxy or by loading a signed bundle with dev mode flags enabled 1), it can be inspected using Chrome DevTools. This provides access to:

  • The IWA shell's JavaScript execution, including breakpoints and stepping.
  • The IWA shell's DOM structure.
  • Network requests made by the IWA shell itself.
  • Application storage (localStorage, IndexedDB) for the IWA shell.
  • Console output from the IWA shell.

6.4. In-App Debug Panel (Optional)

For scenarios where direct DevTools access might be cumbersome or to provide a quick overview, a simple HTML-based debug panel could be integrated into the IWA shell's UI. This panel could display:

  • The current operating mode (IWA / Regular).
  • A list of active tabs with their key states (ID, current URL, <controlledframe> partition ID if applicable).
  • A tail of the most recent log messages from the Logger module. This panel would be toggleable and intended for development/testing builds only.

6.5. Error Handling

Robust error handling will be implemented throughout the application:

  • Asynchronous operations, particularly those involving <controlledframe> methods (which often return Promises 8) and IndexedDB calls, will use try...catch blocks or .catch() handlers for Promises.
  • Failures should be handled gracefully (e.g., displaying an error message to the user or in an error UI within the tab, rather than crashing the shell).
  • Detailed error information, including stack traces where available, will be logged via the DebuggingService.

7. Development Specifications for Coding Agents

These specifications provide a more concrete blueprint for the software agents or developers tasked with implementing the PoC.

7.1. Key JavaScript Modules (Illustrative)

The following modules represent a potential structure. Actual implementation may refine this.

  • main.js:
  • Responsibilities: Application entry point. Detects operating mode (IWA or Regular Web App). Initializes all core singleton modules (e.g., TabManager, NavigationHandler, HistoryStore, PopupHandler, BrowserUI). Sets up global event listeners if necessary.
  • BrowserUI.js:
  • Responsibilities: Manages DOM elements for the main browser shell (address bar, navigation buttons, tab bar container). Handles user interaction events on these static UI elements (e.g., address bar submit, nav button clicks, new tab button click). Communicates user actions to TabManager or NavigationHandler. Renders tab representations in the tab bar based on data from TabManager. Updates UI states (e.g., enabling/disabling nav buttons).
  • Tab.js:
  • Responsibilities: A class or factory function to create Tab objects. Each Tab instance represents an open tab and holds its state: unique ID, current title, current URL, an instance of its ContentView (either ControlledFrameView or MockFrameView), loading status, error status, back/forward capability flags.
  • TabManager.js:
  • Responsibilities: Manages a collection (e.g., an array or map) of Tab instances. Handles requests to create new tabs, activate a specific tab (making it the visible one), and close tabs. Notifies BrowserUI of changes to the tab list or active tab so the UI can re-render.
  • IContentView.js (Interface/Abstract Class):
  • Responsibilities: Defines the common interface that both ControlledFrameView and MockFrameView must implement. This includes methods like load(url), goBack(), goForward(), reload(), stop(), executeScript(details), getCanGoBack(), getCanGoForward(), and properties/event emitters for onLoadCommit, onNewWindow, onConsoleMessage, etc.
  • ControlledFrameView.js:
  • Responsibilities: Implements the IContentView interface using a real <controlledframe> element. Encapsulates all logic for creating, configuring, interacting with, and listening to events from a <controlledframe>.
  • MockFrameView.js:
  • Responsibilities: Implements the IContentView interface using a standard <iframe> element and JavaScript mocking logic as detailed in Section 5.3.
  • ContentViewFactory.js:
  • Responsibilities: A simple factory that, based on the detected operating mode, instantiates and returns either a ControlledFrameView or a MockFrameView when a new content view is needed (e.g., for a new tab or popup).
  • NavigationHandler.js:
  • Responsibilities: Works closely with BrowserUI (for address bar input) and the active Tab's ContentView to initiate navigations. Listens to load-related events from the ContentView to update Tab state (URL, title, loading status) and potentially trigger history logging.
  • HistoryStore.js:
  • Responsibilities: Provides an API for interacting with IndexedDB for storing and retrieving browsing history entries. Encapsulates all IndexedDB setup, transaction, and query logic.
  • PopupHandler.js:
  • Responsibilities: Listens for onNewWindow (or equivalent mocked) events from ContentView instances. When such an event occurs, it orchestrates the creation of a new "popup" view/tab, typically by requesting TabManager to create a new tab with specific properties (e.g., the popup's target URL) and potentially a different UI presentation.
  • Logger.js:
  • Responsibilities: Provides a centralized logging utility (e.g., Logger.info(), Logger.warn(), Logger.error()) used by all other modules. Can be configured for different log levels or output destinations if needed (though console.* is fine for PoC).

7.3. API Interaction Patterns

  • Asynchronous Operations: Promises must be used extensively for all asynchronous API calls. This includes interactions with <controlledframe> methods that return Promises (e.g., back(), executeScript() 8), all IndexedDB operations via HistoryStore, and any Fetch API calls. Async/await syntax should be preferred for readability.
  • Inter-Module Communication: Custom DOM Events, a simple publish/subscribe pattern, or direct method calls with callbacks can be used for communication between the shell's internal modules. For example:
  • TabManager might emit a custom event like 'tabschanged' when the list of tabs is modified, which BrowserUI listens to for re-rendering the tab bar.
  • ContentView instances might emit custom events like 'navigationcommitted' or 'newwindowrequested' that NavigationHandler or PopupHandler listen to.

7.4. Event Flow Diagrams (Conceptual)

  • New Tab Creation:
  1. User clicks '+' button in BrowserUI.
  2. BrowserUI event handler calls tabManager.createAndActivateNewTab(defaultNewUrl).
  3. TabManager calls contentViewFactory.createView() to get a ContentView instance.
  4. TabManager creates a new Tab object, associating it with the new ContentView.
  5. TabManager adds the new Tab to its collection and sets it as active.
  6. TabManager notifies BrowserUI (e.g., via event or callback) about the new tab and active tab change.
  7. BrowserUI updates the DOM to render the new tab in the tab bar and displays the new ContentView's DOM element.
  8. The new ContentView loads its initial URL (e.g., about:blank).
  • URL Load from Address Bar:
  1. User types URL in address bar (managed by BrowserUI) and presses Enter.
  2. BrowserUI captures the input and calls navigationHandler.navigateTo(url).
  3. NavigationHandler retrieves the active Tab instance from TabManager.
  4. NavigationHandler calls activeTab.contentView.load(url).
  5. The ContentView (e.g., ControlledFrameView) sets the src attribute on its underlying <controlledframe> (or <iframe>).
  6. The ContentView listens for load lifecycle events (e.g., onloadstart, onloadcommit, onloadabort).
  7. On onloadstart, ContentView notifies NavigationHandler (or updates Tab state directly) that loading has begun. BrowserUI shows a loading indicator.
  8. On onloadcommit, ContentView notifies NavigationHandler with the final URL and potentially title. NavigationHandler updates the active Tab's state (URL, title, isLoading=false). BrowserUI updates the address bar and tab title. HistoryStore is invoked to log the navigation.
  9. On onloadabort, similar updates for error state.
  • Popup Opening (IWA Mode):
  1. Content within a <controlledframe> (managed by a ControlledFrameView instance) calls window.open('popup.html',...).
  2. The <controlledframe> element emits an onnewwindow event.
  3. The ControlledFrameView instance catches this event.
  4. ControlledFrameView extracts targetUrl and other details from the event and emits a custom newwindowrequested event (or calls a method on) the PopupHandler.
  5. PopupHandler receives the request. It may call event.preventDefault() on the original onnewwindow event object if necessary to stop default browser handling.
  6. PopupHandler instructs TabManager (or a specialized popup manager) to create a new view/tab for the popup, providing the targetUrl. This new view will host its own new <controlledframe>.
  7. The new popup view is displayed within the IWA shell's UI.

7.5. Build Process

  • Development Build (Regular Web App Mode): Standard web development tools like Webpack, Rollup, Parcel, or even simpler setups with ES modules and script concatenation can be used. A local development server (e.g., webpack-dev-server, live-server) will serve the application on http://localhost.
  • Production Build (IWA Mode):
  1. The same build process as above will first generate the static web application assets (HTML, CSS, JS, images).
  2. Web Bundling: Tools like the wbn command-line interface or bundler plugins (e.g., webpack-webbundle-plugin if available and compatible, or custom scripts invoking wbn) will be used to package the built application assets into an unsigned Web Bundle (.wbn file).1
  3. Manifest Inclusion: The manifest.webmanifest file (containing IWA-specific fields like version) must be correctly placed in the source directory (e.g., at the root or in a public folder that gets copied to the build output at /.well-known/manifest.webmanifest relative to the bundle's primary URL) so it's included in the Web Bundle.
  4. Signing: The wbn-sign command-line interface (or equivalent Go tools) will be used to sign the .wbn file with an Ed25519 or ECDSA P-256 private key, producing a Signed Web Bundle (.swbn file).1 The private key needs to be securely managed. The output of wbn-sign includes the Web Bundle ID, which is the IWA's origin.
  5. The resulting .swbn file is what would be deployed or installed into Chrome for testing in IWA mode.

  • Git will be used for version control.

  • src/: Contains all source code (JavaScript modules, CSS, main HTML shell).

8. Conclusion

This specification outlines a feasible path towards developing a proof-of-concept browser leveraging Isolated Web App technology and the <controlledframe> element. The core architecture, centered around a dual-mode development strategy, addresses the practical need for rapid iteration while allowing for exploration of the full IWA capabilities.

The <controlledframe> API provides many of the necessary primitives for emulating browser functionalities such as navigation control, content isolation via partitioning, and popup management through the onnewwindow event. The strict security model of IWAs, including their comprehensive CSP and permission policies, forms a robust foundation, though it also imposes constraints that must be carefully managed (e.g., script sources, network connectivity).

Key challenges and areas for careful attention during development will include:

  • Mocking Fidelity: Creating a sufficiently functional mock of <controlledframe> using <iframe> for the regular web app mode, particularly for features like executeScript across origins and reliable onnewwindow interception.
  • Performance: Managing the resource consumption of potentially multiple process-isolated <controlledframe> instances, especially if many tabs are opened.
  • API Nuances: The <controlledframe> API is relatively new, and its behavior in all edge cases will need to be discovered and handled during development. Access to detailed event object properties for onnewwindow and onloadcommit will be crucial.
  • Debugging Complexity: Debugging interactions between the IWA shell, multiple <controlledframe> instances, and the mocked environment will require diligent use of the outlined debugging strategies.

Despite these challenges, the project is well-positioned to yield valuable insights into the potential of IWAs for creating complex, controlled web content embedding applications. The successful completion of this PoC will demonstrate the extent to which a functional, albeit simplified, browser experience can be recreated within the secure confines of an Isolated Web App.
